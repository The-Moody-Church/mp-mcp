# 2026-04-13 — MCP Server Goes Live

## Summary

Resolved three bugs that prevented Claude Desktop from connecting to the MCP server. The server is now fully operational on TMC1 — Claude Desktop authenticates via MP OIDC and queries MP data through the read-only tools.

## Bugs Fixed

### 1. Missing `expiresAt` in AuthInfo (PR #1)

The MCP SDK's `bearerAuth` middleware requires `expiresAt` (seconds since epoch) in the `AuthInfo` returned by `verifyAccessToken`. The TypeScript type marks it optional, but the runtime throws `InvalidTokenError('Token has no expiration time')` if it's missing. Every `/mcp` request got a 401, causing Claude Desktop to endlessly refresh tokens without ever making a tool call.

**Fix:** Decode the `exp` claim from MP's JWT access token and return it in `AuthInfo`.

### 2. Root path resource metadata (PRs #2, #3)

Claude Desktop sometimes probes `/` instead of `/mcp` when reconnecting. The MCP SDK only serves protected resource metadata at `/.well-known/oauth-protected-resource/mcp`. When Claude Desktop discovered from the root path, it couldn't find the resource metadata (404), and after auth completed it wouldn't attach the Bearer token to root-path requests due to resource URL scoping.

**Fix:** Mount MCP handlers at both `/` and `/mcp`. Serve protected resource metadata at `/.well-known/oauth-protected-resource` pointing to root as the resource, with a separate `bearerAuth` instance referencing the root metadata URL.

### 3. Auth info nesting in tool handlers (PR #4)

`getAuthFromExtra()` looked for `mpBaseUrl` and `accessToken` at `authInfo.mpBaseUrl`, but they're nested inside `authInfo.extra.mpBaseUrl` (the `AuthInfo.extra` property set by `verifyAccessToken`). All tool calls that hit the MP API failed with "Not authenticated".

**Fix:** Read from `authInfo.extra.mpBaseUrl` and `authInfo.extra.accessToken`.

## Observations for Next Session

- **No raw IDs:** Claude should omit Contact_ID, Participant_ID, etc. unless explicitly asked
- **Lookup values:** Claude guessed Marital_Status_ID=1 as "married" (it's "single"). Needs lookup resolution
- **Donation sensitivity:** Need a policy for how Claude discusses donor/giving data
- **Count tool:** `query_table` doesn't support COUNT — Claude had to paginate to count 43k contacts. A `count_records` tool would help
- **Debug logging:** HTTP request and MCP handler logging still in `index.ts` — remove once stable
